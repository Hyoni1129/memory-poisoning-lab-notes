# memory-poisoning-lab-notes

Research on memory poisoning for the 2026 CURT Undergraduate Research Program.

## Purpose
This repository captures the research process, not just final artifacts.

Use it to preserve:
- daily work logs
- decision rationale
- literature and paper notes
- deep research reports (including LLM-assisted research)
- experiment commentary
- failures and lessons learned
- meeting notes
- evolving research plans and ideas

## Relationship To Main Research Repository
Companion formal repository:
- `memory-poisoning-research`
- Planned GitHub URL: `https://github.com/<your-username>/memory-poisoning-research`
- Local path (sibling repo): `../memory-poisoning-research`

Separation principle:
- `memory-poisoning-research`: reproducible code/results/paper artifacts
- `memory-poisoning-lab-notes`: reasoning, logs, decisions, and reflective context

## Organization Philosophy
- Keep one clear purpose per folder.
- Prefer many short, focused notes over long mixed documents.
- Use shared experiment IDs to connect notes to formal artifacts.
- Keep entries concrete: what changed, why it changed, what is next.

## Repository Structure
- `00_research_plan/`: goals, milestones, and plan evolution
- `01_background/`: concepts and paper understanding
- `02_decision_log/`: explicit decisions and alternatives
- `03_daily_log/`: one file per day/session
- `04_deep_research_reports/`: generated deep reports with verification status
- `05_experiment_notes/`: informal commentary linked to experiment IDs
- `06_failures/`: failure narratives and lessons learned
- `07_meetings/`: advisor or collaborator meeting notes
- `08_reading_list/`: reading queue and paper status tracking
- `09_ideas/`: idea backlog
- `templates/`: reusable note templates
- `docs/`: operating guides and policies

## Consistency Rules
1. Use `YYYY-MM-DD` in date-based filenames.
2. Use shared experiment IDs (for example, `exp_002_poison_authority`).
3. Follow note naming rules in `docs/note-naming-conventions.md`.
4. Follow commit style in `docs/commit-message-guide.md`.

## Start Here
1. Read `SETUP_SUMMARY.md`.
2. Review `docs/README.md`.
3. Create your first daily log from `templates/daily-log-template.md`.
4. Add decisions as soon as they are made, not later.
