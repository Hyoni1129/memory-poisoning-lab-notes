# LLM 에이전트에서 간접 프롬프트 인젝션과 메모리 포이즈닝이 작동하는 방식

## 요약

이 보고서는 업로드된 과제 지시문에 맞추어, 실험 설계가 아니라 **공격 메커니즘의 이해**에 초점을 둔 두 번째 준비 보고서로 작성되었다. 독자는 LLM의 기본 개념은 알고 있지만 에이전트 보안과 공격 분류에는 익숙하지 않다는 가정을 둔다. 또한 최근 연구 흐름을 반영하기 위해 2024–2026 자료를 우선 사용했다. fileciteturn0file0

핵심 결론은 단순하다. LLM 에이전트 공격의 중심에는 “외부에서 들어온 텍스트가 어떻게 모델에게 **명령처럼 읽히는가**”라는 문제가 있다. 일반 소프트웨어는 코드와 데이터를 문법적으로 구분하지만, LLM은 시스템 지시, 사용자 입력, 웹페이지 본문, 로그, 첨부 문서, 도구 출력 등을 모두 같은 토큰 공간에서 처리한다. 그래서 공격자는 외부 콘텐츠 안에 지시문 같은 텍스트를 심고, 에이전트가 그것을 읽게 만들고, 도구를 호출하게 하거나, 그 내용이 메모리에 저장되도록 유도할 수 있다. 이 지점에서 프롬프트 인젝션은 단순한 채팅 실패가 아니라, **행동 가능한 시스템을 오작동시키는 구조적 취약점**이 된다. citeturn30view4turn22search0turn16view4turn15view6

간접 프롬프트 인젝션은 직접 프롬프트 인젝션보다 더 위험할 수 있다. 이유는 공격자가 사용자의 입력창에 직접 들어오지 않아도 되기 때문이다. 웹페이지, 이메일, PDF, 로그, 도구 설명, 검색 문서, 심지어 접근성 트리나 렌더링된 웹 요소 속에 숨겨진 명령이 에이전트에 의해 읽히면, 공격은 “데이터를 처리하는 과정” 속에서 실행된다. 최근 연구는 공개 웹페이지에 심은 명령이 웹 에이전트를 높은 비율로 오염시킬 수 있음을 보였고, 클라우드 로그를 읽는 디버깅 에이전트가 로그 내 명령을 실행하거나 원격 코드 실행으로 이어질 수 있음도 보여주었다. citeturn16view1turn31view0turn15view4turn15view5

메모리 포이즈닝은 이 문제를 한 단계 더 심각하게 만든다. 간접 인젝션이 “지금 이 세션에서의 오염”이라면, 메모리 포이즈닝은 “나중 세션까지 남는 오염”이다. 최근 연구들은 공격자가 메모리 저장소에 직접 접근하지 않아도, 정상적인 상호작용만으로 악성 기록을 메모리에 심고, 이후의 질의에서 그 기록이 검색되어 다시 프롬프트로 주입되도록 만들 수 있음을 보여준다. 특히 eTAMP 계열 연구는 **환경만 조작**해도 에이전트가 스스로 오염된 관찰을 저장하고, 다른 웹사이트와 다른 세션에서까지 그 영향이 이어질 수 있음을 보여준다. citeturn15view7turn17view0turn18view2turn18view3turn35search3

학생의 연구 주제인 “instruction-like malicious payload가 memory consolidation을 거쳐 살아남고, 이후 검색되어 행동에 영향을 줄 수 있는가”는 매우 타당하다. 현재 연구는 요약과 압축이 무엇을 남기고 무엇을 버리는지, 검색이 어떤 표현을 다시 살려내는지, 그리고 악성 지시문과 정상적인 작업 기억을 어떻게 구분할지를 아직 완전히 해결하지 못했다. 따라서 **요약 생존성**, **검색 재활성화**, **악성과 정상의 구분 문제**를 분리해 보는 접근은 의미가 크다. citeturn34view0turn33view0turn17view4turn34view4turn34view5

## 문제의 출발점

### 프롬프트 인젝션이 생기는 이유

직관적으로 말하면, LLM은 “이 문장은 읽어야 할 데이터인지”와 “이 문장은 따라야 할 명령인지”를 전통적 프로그램처럼 명확한 구문 규칙으로 구분하지 못한다. 사람은 웹페이지 본문과 브라우저의 시스템 설정을 다르게 느끼지만, 모델은 둘 다 토큰 시퀀스로 본다. 그래서 웹페이지 속 문장이 충분히 “명령처럼 들리면” 모델은 그것을 사용자나 시스템의 의도로 오인할 수 있다. citeturn30view4turn16view4

기술적으로 보면 이 문제는 **instruction-data separation의 부재**로 설명된다. 최근 연구는 현재 LLM이 “실행해야 할 지시”와 “처리해야 할 데이터”를 명시적으로 분리하지 못하며, 이 때문에 간접 프롬프트 인젝션 같은 조작에 취약하다고 지적한다. 같은 맥락에서 entity["organization","OWASP","appsec nonprofit"]의 최신 가이드는 프롬프트 인젝션을 LLM 애플리케이션의 최상위 위험으로 두고, 자연어 지시와 데이터가 명확한 경계 없이 함께 처리된다는 점을 취약성의 핵심으로 설명한다. citeturn22search0turn15view2turn30view4

이 문제는 전통적인 소프트웨어 보안과도 다르다. 전통적 시스템에서는 코드와 데이터가 파서, 타입 시스템, 실행 권한, API 스키마 같은 계층으로 분리된다. 반면 LLM 시스템에서는 “권위 있는 말투”, “지시문 같은 형식”, “역할을 흉내 내는 표현”이 모델 내부에서 실제 권한과 비슷하게 인식될 수 있다. 2026년 연구는 이를 **role confusion**으로 설명하며, 텍스트가 실제로 어디서 왔는지보다 “어떤 역할처럼 들리는지”가 공격 성공을 더 잘 예측한다고 보였다. citeturn16view3turn16view4

이 때문에 프롬프트 인젝션은 단순한 입력 검증 문제를 넘어, 더 근본적인 **권한 혼동(confusable deputy)** 문제로 보는 것이 맞다. entity["organization","National Cyber Security Centre","uk cybersecurity agency"]는 프롬프트 인젝션을 고전적 코드 인젝션으로 보기보다, 본질적으로 혼동되기 쉬운 대리자를 악용하는 문제로 보아야 한다고 설명한다. 즉, 에이전트는 공격자의 텍스트를 읽었을 뿐인데, 자신이 가진 더 높은 권한으로 행동하게 된다. citeturn15view6

### 직접 프롬프트 인젝션과 간접 프롬프트 인젝션

직접 프롬프트 인젝션은 공격자가 사용자 입력 안에 노골적인 지시를 넣는 형태다. 예를 들어 “이전 지시를 무시하라” 같은 문구가 여기에 해당한다. 반대로 간접 프롬프트 인젝션은 웹페이지, 문서, 이메일, 로그, 파일 같은 **외부 콘텐츠** 안에 지시를 숨겨 두고, 에이전트가 그 콘텐츠를 읽도록 만들어 발생한다. OWASP 문서는 직접 인젝션을 사용자의 프롬프트가 모델 동작을 바꾸는 경우로, 간접 인젝션을 외부 स्रोत의 입력이 모델 동작을 바꾸는 경우로 구분한다. citeturn16view0turn30view4

간접 인젝션이 더 위험한 이유는 세 가지다. 첫째, 공격이 “입력창”이 아니라 사용자가 신뢰하는 데이터 경로를 통해 들어온다. 둘째, 공격 텍스트가 웹 본문, HTML 요소, 도구 출력, 로그 메시지처럼 **데이터로 위장**된다. 셋째, 한 번 오염된 콘텐츠는 여러 사용자가 반복해서 읽게 될 수 있어 공격이 자연스럽게 확산된다. 2023년 고전적 간접 인젝션 연구는 검색된 데이터 안에 심어진 지시문이 원격으로 LLM 통합 애플리케이션을 장악할 수 있음을 보였고, 이후 연구들은 이를 에이전트 환경으로 확장했다. citeturn20search0turn15view0turn31view0

웹 에이전트 분야에서는 이 위험이 특히 뚜렷하다. WIPI는 공개 웹페이지에 심은 악성 지시가 블랙박스 환경에서도 웹 에이전트를 높은 성공률로 조종할 수 있음을 보였고, HTML 접근성 트리를 이용한 2025년 연구는 실제 웹사이트에서 로그인 자격 증명 탈취나 강제 광고 클릭 같은 공격이 가능함을 보여주었다. 또 WASP 벤치마크는 고급 추론 모델을 사용한 웹 에이전트도 현실적인 시나리오에서 낮은 비용의 인간 작성 인젝션에 속을 수 있음을 보였다. citeturn31view0turn15view4turn31view1

간접 인젝션은 웹페이지만의 문제가 아니다. LogJack 연구는 클라우드 로그를 읽고 조치 명령을 내리는 디버깅 에이전트가 로그 안의 지시문에 영향을 받아 명령을 실행할 수 있음을 보였으며, 심지어 `curl | bash` 형태의 원격 코드 실행이 8개 모델 중 6개에서 성공했다고 보고했다. 더 중요한 점은, AWS·GCP·Azure 계열의 보호 장치가 같은 페이로드를 고립된 문장으로는 탐지해도, **로그 문맥 안에 섞이면 잘 탐지하지 못했다**는 사실이다. citeturn15view5

## 에이전트 공격이 작동하는 경로

### 주입에서 행동까지의 전체 파이프라인

에이전트 공격은 보통 다음 흐름으로 이해하면 된다.

> 외부 환경에 악성 텍스트 삽입 → 에이전트가 이를 관찰하거나 도구 출력으로 수신 → 모델 컨텍스트에 편입 → 모델이 지시로 해석 → 도구 호출 또는 행동 실행 → 기록이 메모리에 저장 → 후속 세션에서 검색되어 다시 컨텍스트로 재주입

이 파이프라인에서 가장 중요한 단계는 **ingestion**, 즉 “에이전트가 그것을 읽는 순간”이다. 악성 문장이 웹상에 존재하기만 해서는 충분하지 않다. 그것이 검색되거나, 요약되거나, 도구 출력으로 반환되거나, 브라우저 에이전트의 관찰 상태에 들어와야만 실제 위험으로 바뀐다. 2026년 SoK는 현실적 에이전트가 환경 관찰을 바탕으로 행동하는 **context-dependent task**를 중요하게 보아야 한다고 지적했는데, 바로 이 지점에서 ingestion이 공격 표면이 된다. citeturn15view0turn25view0turn25view3

에이전트 시스템은 일반 채팅 모델보다 위험을 증폭한다. 채팅 모델은 보통 텍스트를 내보내는 데서 끝나지만, 에이전트는 이메일을 보내고, 웹을 클릭하고, 파일을 수정하고, API를 호출하고, 다음 단계 계획을 세운다. Agent Security Bench는 에이전트가 도구와 메모리를 사용하면서 시스템 프롬프트, 사용자 프롬프트 처리, 도구 사용, 메모리 검색 등 여러 단계에서 취약해질 수 있다고 보고했고, The Attack and Defense Landscape of Agentic AI는 비에이전트 LLM 위험이 에이전트 맥락에서는 데이터 유출과 시스템 조작으로 증폭된다고 정리한다. citeturn14view3turn24view4

### 메모리 포이즈닝의 핵심 개념

메모리 포이즈닝은 악성 콘텐츠가 **지속 메모리**에 저장되고, 나중의 질의나 작업에서 다시 검색되어 미래 행동을 바꾸는 공격이다. entity["organization","OWASP","appsec nonprofit"]의 AI Agent Security 치트시트는 이를 “미래 세션이나 다른 사용자에게 영향을 주기 위해 에이전트 메모리에 악성 데이터가 지속되는 것”으로 설명한다. OWASP Agent Memory Guard도 런타임에 쓰기 가능한 지속 메모리가 세션을 넘나드는 misalignment, data exfiltration, malicious behavior의 고가치 공격 표면이 된다고 본다. citeturn28view0turn28view1

프롬프트 인젝션과 메모리 포이즈닝의 차이는 **시간 축**에 있다. 일반적인 프롬프트 인젝션은 현재 컨텍스트 창 안에서 끝나는 경우가 많지만, 메모리 포이즈닝은 현재 세션이 끝난 뒤에도 남는다. 장기 메모리 보안 연구는 단발성 인젝션과 달리, 오염된 메모리 항목은 이후의 수많은 작업에서 반복해서 호출될 수 있다고 지적한다. MINJA와 AgentPoison은 각각 질의만으로 메모리를 오염시키거나, 장기 메모리와 RAG 지식베이스를 오염시켜 뒤늦게 행동을 바꾸는 구체적 공격을 제시했다. citeturn35search3turn15view7turn17view0

지속성은 위협을 더 교묘하게 만든다. 공격이 주입되는 순간과 실제 피해가 발생하는 순간이 분리되기 때문이다. 한 세션에서는 단지 “이상한 글”을 저장했을 뿐처럼 보이지만, 나중 세션에서 관련 질의가 들어오면 그것이 검색되어 행동을 오염시킬 수 있다. 이 지연성 때문에 원인 추적이 어려워지고, 보안팀은 실제 행동이 발생했을 때 이미 원래 주입 지점을 잃어버렸을 수 있다. citeturn18view3turn35search3

### 환경 주입형 공격

환경 주입형 공격에서 중요한 점은 공격자가 메모리 DB를 직접 수정하지 않는다는 것이다. 대신 웹페이지, 상품 설명, 로그, 이메일, 관찰 가능한 화면 같은 **환경 자체**를 바꾼다. 에이전트는 그 환경을 합법적 입력으로 읽고, 그 내용을 작업 기록이나 반성(reflection), 과거 경험, 벡터 저장소 등에 저장한다. 여기서 공격은 “메모리에 침투”한 것이 아니라, 에이전트의 정상 학습·기억 메커니즘을 악용한 것이다. citeturn18view2turn28view1

eTAMP는 이 점을 매우 잘 보여 준다. 이 연구는 공격자가 환경 관찰만 오염시켜도, 에이전트가 그것을 메모리에 저장한 뒤 **다른 세션, 다른 사이트**에서 재활성화될 수 있음을 보였다. 논문은 이것을 “direct memory access 없이 가능한 cross-session, cross-site compromise”로 설명하고, 단 한 번의 오염된 관찰만으로도 이후 과업에서 공격이 활성화될 수 있다고 보고한다. citeturn18view0turn18view1turn18view5

이 유형이 특히 현실적인 이유는 실제 서비스에서 메모리 저장소는 종종 접근 제어나 인증으로 보호되기 때문이다. 그러나 에이전트가 보는 웹 환경, 로그, 문서, 도구 출력은 본질적으로 훨씬 더 넓은 공격 표면을 가진다. eTAMP는 이런 환경 조작이 권한 기반 방어를 우회할 수 있음을 보여 주었고, 환경 스트레스가 걸릴 때 공격 성공률이 최대 8배까지 증가할 수 있다고 보고했다. 즉, 성능이 떨어지는 순간은 곧 보안이 떨어지는 순간일 수 있다. citeturn18view3turn18view4

### 메모리와 검색이 공격을 키우는 이유

메모리는 공격을 단지 “보관”하는 것이 아니라, 다시 **활성화**할 수 있게 만든다. entity["company","Microsoft","technology company"]의 Agent Framework 문서는 채팅 히스토리 메모리 제공자가 과거 메시지를 벡터 저장소에 넣고, 이후 현재 입력과 의미적으로 유사한 기록을 검색하여 **다시 현재 컨텍스트에 주입**한다고 설명한다. 이것이 바로 retrieval이 왜 reactivation mechanism인지 보여 준다. 기억은 저장만으로 끝나지 않고, 새로운 프롬프트 안으로 다시 들어온다. citeturn15view8turn17view4

그래서 “포이즈닝이 성공했다”와 “공격이 발화했다”는 같은 말이 아니다. 공격은 저장 후에도 검색 단계를 통과해야 한다. AgentPoison은 아예 악성 예시가 검색되도록 트리거를 최적화하고, 질의가 같은 최적화 트리거를 포함할 때 악성 시연이 검색되도록 설계한다. MINJA도 마찬가지로 후속 피해자 질의에서 악성 기록이 쉽게 검색되도록 bridging step과 progressive shortening을 사용한다. 즉, **검색되지 않으면 발화하지 않는다**는 점이 메모리 공격의 핵심이다. citeturn17view0turn21view3turn21view4

유사도 검색이 중요하다는 말은, 공격이 정확한 문자열 일치에만 의존하지 않는다는 뜻이다. 현재 메모리 시스템은 보통 의미적 유사성으로 기록을 찾는다. 그래서 공격자는 문구 자체보다 **질의와 의미적으로 다시 만날 가능성**을 노린다. AgentPoison이 트리거 질의를 임베딩 공간의 별도 영역으로 몰아넣어 검색률을 높이고, CMA 연구가 연관 검색·다중 홉 recall·선택적 유지가 미래 검색에 영향을 준다고 설명하는 이유도 여기에 있다. 학생 연구에서 paraphrase query를 넣어 보는 것이 중요한 이유가 바로 이것이다. 이는 검색 메커니즘상 매우 합리적인 가설이다. citeturn17view2turn34view4turn34view5

또 하나 중요한 점은 retrieval이 단지 과거를 불러오는 것이 아니라, 미래 접근성을 다시 바꾼다는 사실이다. CMA 연구는 반복적으로 조회된 조각이 더 안정화되고, 모순되는 조각은 약화된다고 설명한다. 이를 보안 관점에서 읽으면, 한 번 재활성화된 악성 기억은 그 자체로 다음 재활성화를 더 쉽게 만들 수도 있다. 이 지점이 “일시 오염”이 “지속 행동 편향”으로 바뀌는 중요한 연결고리다. citeturn34view4

## 방어가 어려운 이유와 흔한 오해

### 방어가 어려운 이유

첫째 난점은 경계가 애초에 불명확하다는 것이다. 모델은 신뢰된 시스템 지시와 비신뢰 사용자 입력을 한꺼번에 처리해야 하며, 공격자는 바로 그 혼합 지점에서 권한을 가로챈다. NCSC는 이를 confusable deputy 관점에서 설명했고, role confusion 연구는 모델이 실제 태그보다 역할을 흉내 내는 스타일 신호에 더 끌릴 수 있음을 보여 주었다. 따라서 “지침을 더 세게 써라” 또는 “태그를 더 잘 붙여라”만으로는 충분하지 않다. citeturn15view6turn16view4

둘째 난점은 공격이 너무 자연스럽게 보일 수 있다는 점이다. 간접 인젝션은 스팸성 문자열이 아니라, 정상적인 웹카피, 도움말 문서, 로그 메시지, 반성 문장, 도구 설명, 작업 메모 안에 섞인다. WebSentinel이 웹페이지를 “untrusted execution environment”로 보아야 한다고 주장하는 이유도 여기에 있다. 공격 텍스트가 지저분하거나 이상해서가 아니라, 너무 그럴듯해서 탐지가 어렵다. citeturn31view3

셋째 난점은 공격이 다단계라는 점이다. 입력 시점에는 주입이고, 중간에는 저장이며, 나중에는 재활성화다. 최근 SoK들은 기존 벤치마크가 이런 context-dependent task를 과소평가해 왔다고 비판한다. 실제 에이전트는 환경을 읽어 행동해야 하므로, 외부 맥락을 강하게 차단하면 보안은 나아질 수 있어도 유틸리티가 무너질 수 있다. 그래서 “정확도·유용성·보안”을 동시에 만족시키는 단일 방어가 어렵다. citeturn25view0turn26view2

넷째 난점은 전통적 필터와 이상 탐지가 자주 실패한다는 점이다. Zhan 등은 간접 프롬프트 인젝션 방어 8종을 adaptive attack으로 모두 우회했고, 2026년 prompt injection defense evaluation은 모델이 스스로 자신을 지키는 방식의 방어가 결국 모두 무너졌다고 보고했다. 다만 이 후자 연구는 주로 비밀 유출 setting에서 **응답단 output filtering**이 강하다는 것을 보였는데, 이는 특정 문제에 강한 것이지 모든 agentic action을 일반적으로 해결했다는 뜻은 아니다. 오히려 중요한 교훈은 보안 경계를 모델 내부가 아니라 **애플리케이션 코드와 실행 계층**에 두어야 한다는 점이다. citeturn13search0turn24view6

다섯째 난점은 공격이 “행동”보다 먼저 “믿음”을 오염시킬 수 있다는 것이다. 최근 연구는 poisoned memory entries가 시간이 지나며 agent’s belief state를 재형성할 수 있다고 말한다. 이 관점은 중요하다. 공격은 반드시 즉시 악성 동작을 만들 필요가 없다. 먼저 “이 사용자가 이런 걸 원한다”, “이 규칙이 더 우선이다”, “이 상황에서는 이 도구를 쓰는 게 맞다” 같은 내부 작업 가정을 바꾸고, 이후의 정상적 의사결정 경로를 틀어 놓을 수 있다. 바로 이 점 때문에 anomaly detection이 더 어려워진다. 외형상 행동이 정상 워크플로우처럼 보이기 때문이다. citeturn35search2turn35search10turn35search7

### 학생 연구 질문과의 연결

학생의 첫 번째 질문인 **RQ1: payload가 summarization을 거쳐 살아남는가**는 타당하다. 최근 메모리 연구는 요약이 오래된 대화와 실행 기록을 압축할 때 “decision-relevant information”을 남기는 경향이 있다고 설명한다. 따라서 악성 payload가 단순한 무작위 잡음이 아니라, 목표·규칙·주의사항·제약처럼 **결정 관련 정보처럼 보이면** 요약에서 살아남을 가능성이 있다. 동시에 별도의 메모리 연구는 summarization/compaction이 많은 사실을 잃게 하고 프로젝트 제약도 상당 부분 침식시킬 수 있다고 보고했다. 즉, 생존 여부는 자동으로 보장되지 않으며, 바로 그 때문에 실험 가치가 있다. citeturn34view0turn33view0

두 번째 질문인 **RQ2: retrieval이 왜 activation에 필수인가**도 매우 잘 설정된 질문이다. 현대 에이전트 메모리는 저장된 기록을 현재 입력과 의미적으로 맞는 경우에만 다시 컨텍스트로 넣는 경우가 많다. AgentPoison과 MINJA가 모두 “검색되기 쉬운 악성 기록”을 설계하는 데 집중하는 이유가 여기에 있다. 따라서 학생의 실험에서 payload를 심어 놓고, 이후 질의를 바꿔 가며 검색될 때와 검색되지 않을 때를 비교하는 것은 공격 메커니즘을 정면으로 건드리는 올바른 방식이다. citeturn15view8turn17view0turn21view3

세 번째 질문인 **RQ3: benign retention과 malicious retention을 어떻게 구분할 것인가**는 연구적으로 더 어렵지만 더 중요하다. 정상 에이전트는 사실상 “유용한 규칙, 제약, 사용자 선호, 작업 히스토리”를 기억해야 한다. CMA 연구가 selective retention을 통해 최신·관련 정보를 남기고 오래된 정보를 약화시켜야 한다고 말하는 이유도 यही다. 따라서 학생의 과제는 “무엇이 기억되느냐”보다 “어떤 **형태의 지시성**이 기억되고, 어떤 것은 버려지느냐”를 분리하는 것이다. 이때 control group에서 directive cues를 제거하는 설계는 매우 중요해진다. citeturn34view4turn28view1

이런 맥락에서 **pseudo trajectory experiment**는 충분히 의미 있다. 현실의 메모리 포이즈닝 연구도 종종 정상적인 다단계 상호작용, reflection, browsing trajectory, tool trace가 메모리로 쓰이는 과정을 활용한다. eTAMP는 환경 관찰 기반 trajectory memory poisoning을, MINJA는 상호작용만으로 메모리 주입을 보여 주었다. 따라서 학생이 “한 번의 악성 문자열”이 아니라, 에이전트가 실제로 경험하고 요약하고 저장할 법한 **유사 trajectory**를 구성하는 것은 기계적 타당성이 있다. citeturn18view2turn21view4

### 페이로드 설계에 주는 시사점

이 부분은 공격 레시피가 아니라, 학생이 **가설을 세울 때 어떤 속성을 변수로 봐야 하는지**를 설명하는 수준으로 이해하는 것이 안전하고 적절하다. 현재 문헌을 종합하면, summary에 살아남고 이후 retrieval에 걸릴 가능성이 높은 payload는 보통 우연한 잡음이 아니라 **결정 관련처럼 보이는 구조**를 가진다. 요약 시스템은 decision-relevant information을 남기려 하고, role confusion 연구는 권위 있는 역할을 흉내 내는 표현이 모델 내부에서 더 큰 영향력을 갖는다고 말한다. 따라서 “명령처럼 읽히는 형식”이 생존성과 영향력을 높일 가능성이 있다는 가설은 문헌적으로 뒷받침된다. 다만 이것은 일반 법칙이 아니라, 현재까지의 메모리·요약·검색 구조를 근거로 한 강한 연구 가설이다. citeturn34view0turn16view4

학생 관점에서 보면, 살아남기 쉬운 요인은 대체로 네 가지 축으로 정리할 수 있다. **authority tone**은 내용을 규칙이나 정책처럼 보이게 만들고, **instruction-like phrasing**은 데이터가 아니라 실행 지침처럼 읽히게 만들며, **repeated structure**는 요약 과정에서 gist로 압축되기 쉽게 만들고, **semantic relevance**는 나중 질의와의 유사도 검색 가능성을 높인다. AgentPoison이 임베딩 공간 자체를 조작해 retrieval을 높이려 하고, 메모리 시스템이 의미 유사성으로 기록을 찾는다는 점은 이 직관을 강화한다. citeturn17view2turn15view8turn34view5

반대로 실패하기 쉬운 페이로드는 대체로 **무작위 텍스트**, **업무와 무관한 잡음**, **약한 구조**, **검색 질의와 멀리 떨어진 의미**를 가진다. 요약은 모든 것을 동일하게 남기지 않고, 검색은 모든 저장 항목을 같은 확률로 불러오지 않는다. 즉, “저장되었다”는 사실만으로 충분하지 않다. 생존성과 재활성화는 각각 별개의 관문이다. citeturn34view0turn15view8

이 때문에 control group은 단순히 benign text를 넣는 데서 끝나면 안 되고, **directive cue를 제거한 instruction-like vs non-instruction-like 비교**가 되어야 한다. 그래야 살아남음의 원인이 “문장이 길어서”, “반복됐어서”, “주제와 관련 있어서”인지, 아니면 정말로 “지시처럼 보였기 때문인지”를 분리할 수 있다. 학생이 이 통제에 실패하면, 나중에 교수에게 설명할 때 “메모리가 악성을 기억한 것인지, 그냥 salient한 문장을 기억한 것인지”를 구분할 수 없게 된다. 이 점은 특히 중요하다. citeturn34view0turn34view4turn16view4

## 결론과 다음 읽을거리

### 흔한 오해

“인젝션은 곧바로 실행되는 공격이다”라는 생각은 틀리다. 최근 메모리 포이즈닝 연구들은 공격이 한 세션에서는 저장만 되고, 다른 세션에서 검색될 때 비로소 활성화될 수 있음을 보여 준다. 특히 eTAMP는 단일 오염 관찰이 이후의 다른 웹사이트 작업에서 발화할 수 있음을 보였다. citeturn18view3turn35search3

“메모리는 내부니까 안전하다”라는 생각도 틀리다. MINJA는 공격자가 메모리 저장소를 직접 수정하지 않아도 질의만으로 악성 기록을 심을 수 있음을 보였고, eTAMP는 환경만 조작해도 메모리 오염이 가능하다고 보였다. 내부라는 사실은 곧 안전을 뜻하지 않는다. 오히려 런타임에 쓰기 가능한 내부 상태라는 점이 핵심 위험이다. citeturn15view7turn18view2turn28view1

“모델이 한 번 악성 지시를 무시했으니 안전하다”도 오해다. adaptive attack 연구는 고정된 테스트셋에서 잘 버티던 방어가 적응형 공격 앞에서는 쉽게 무너질 수 있음을 보여 주었다. 게다가 기억 기반 공격은 한 번의 실패 후 끝나는 것이 아니라, 다른 표현·다른 세션·다른 검색 경로로 다시 시도될 수 있다. citeturn13search0turn24view6

“입력 필터링만 잘하면 된다” 역시 틀리다. OWASP 치트시트는 공격 유형 자체가 direct, remote/indirect, HTML/Markdown, persistent, RAG poisoning, agent-specific attack 등으로 넓다고 정리한다. 2026년 defense evaluation은 input-side 혹은 model-reliant defense가 결국 무너질 수 있음을 보였고, LogJack은 문맥 속에 숨은 로그 인젝션이 guardrail에 잘 잡히지 않음을 보여 주었다. 입력 필터는 필요하지만 충분조건이 아니다. citeturn30view0turn30view1turn30view2turn30view3turn24view6turn15view5

### 초심자 연구자를 위한 결론

초심자 연구자가 가장 먼저 이해해야 할 것은, 에이전트 공격의 핵심 메커니즘이 “악성 문자열 그 자체”가 아니라 **외부 환경의 텍스트가 권위 있는 컨텍스트로 승격되는 과정**이라는 점이다. 그 순간 모델은 데이터를 처리하는 것이 아니라, 공격자가 만든 의도를 실행하게 될 수 있다. 에이전트에서는 이 승격이 도구 호출, 계획, 메모리 기록, 후속 검색과 연결되므로 피해가 더 커진다. citeturn15view0turn24view4turn28view0

메모리 포이즈닝이 중요한 연구 주제인 이유는 공격의 시간을 늘리기 때문이다. 단발성 인젝션은 현재 창 안에서 끝날 수 있지만, 메모리 포이즈닝은 미래의 세션, 미래의 질의, 심지어 다른 사이트에서의 작업으로 영향이 확장될 수 있다. 지속성은 곧 은닉성이고, 은닉성은 곧 탐지 난이도 상승이다. citeturn35search3turn18view0

환경 주입형 공격이 특히 중요한 이유는 현실적이기 때문이다. 공격자는 메모리 데이터베이스를 해킹하지 않아도 된다. 사용자가 읽을 수 있는 페이지, 에이전트가 방문할 수 있는 사이트, 시스템이 수집하는 로그, 또는 도구가 반환하는 텍스트만 조작하면 된다. 이것은 실제 배포 환경에서 훨씬 더 가능한 위협 모델이다. citeturn18view2turn15view5

다음 보고서로 넘어가기 전에 학생이 반드시 이해해야 할 것은 세 가지다. 첫째, **survival**은 요약과 저장의 문제다. 둘째, **activation**은 검색과 재주입의 문제다. 셋째, **harm**는 행동이 아니라 종종 belief state의 오염에서 시작된다는 점이다. 이 세 층위를 분리해서 생각할 수 있어야, 이후 실험 설계가 “그냥 이상한 현상 관찰”이 아니라 명확한 메커니즘 검증이 된다. citeturn34view0turn15view8turn35search10

### 열린 질문과 한계

현재 문헌의 상당수는 2025–2026년의 최신 preprint와 벤치마크에 기반하고 있으며, 실제 상용 에이전트의 메모리 write policy, summarization policy, approval flow는 시스템마다 크게 다르다. 또한 어떤 방어가 잘 되는지는 비밀 유출, 웹 행동 조작, 도구-stream 오염, 장기 기억 오염 등 **어떤 위협을 측정하느냐**에 따라 달라질 수 있다. 따라서 학생의 연구는 “모든 에이전트에 대한 보편 법칙”을 주장하기보다, 특정 memory pipeline에서 survival과 retrieval이 어떻게 작동하는지 보여 주는 방식이 더 설득력 있다. citeturn24view4turn25view0turn24view6

- 프롬프트 인젝션은 LLM이 지시와 데이터를 명확히 분리하지 못하기 때문에 생기는 구조적 취약점이다. citeturn22search0turn30view4
