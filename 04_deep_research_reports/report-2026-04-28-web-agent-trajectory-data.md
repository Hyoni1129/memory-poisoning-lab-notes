공부 진행용DeepResearch_Part 5_(Chat_GPT)

# 웹 에이전트 궤적 데이터: 수집·구조·전처리 심층 연구 보고서

## 요약  
웹 에이전트 연구에서 *궤적 데이터*는 에이전트의 관측, 행동, 추론(chain-of-thought) 정보를 모두 담아야 하며 이를 통해 복잡한 작업 수행을 평가할 수 있다. 본 보고서는 WebArena 등 최신 웹 환경 벤치마크를 중심으로, 에이전트 궤적 데이터를 어떻게 수집·구조화하고 전처리할 것인지를 다룬다. WebArena 프로젝트에서 이미 179개 작업에 대한 사람 궤적이 Playwright trace 파일로 공개되어 있으며【14†L259-L266】, 각 궤적에는 화면 캡처, 접근성 트리(accessibility tree), 에이전트의 추론 로그(raw prediction)와 파싱된 행동(parsed action)이 포함된다【14†L298-L305】. 이처럼 풍부한 데이터를 체계적으로 처리하려면 행동 정규화, 성공/실패 라벨링, 중복 제거 등이 필요하다【1†L227-L234】【1†L236-L240】. 동시에 데이터 품질과 편향, 누수 등 문제를 주의해야 한다【19†L112-L118】【29†L110-L115】. 따라서 본 보고서는 웹 에이전트 궤적 데이터의 주요 속성과 수집 방법, 데이터 구조 설계, 전처리 절차, 일반적인 함정 및 권장사항을 체계적으로 제시한다.  

## 1. 연구 질문  
- **핵심 질문**: 웹 에이전트의 궤적 데이터를 수집·구조화·전처리하기 위한 최적 방법은 무엇인가?  
- **상세 질문**: (1) WebArena 등 웹 에이전트 환경에서 데이터 수집을 위해 필요한 절차는? (2) 궤적 데이터는 관측(observation), 행동(action), 추론(reasoning)으로 어떻게 구조화해야 하는가? (3) 데이터 품질(issue)과 다양성 측면에서 고려사항은 무엇인가? (4) 제어된 실험을 위해 데이터 전처리(design, canonicalization, 라벨링)는 어떻게 설계해야 하는가? (5) 실무에서 흔히 범할 수 있는 실수와 그 예방책은 무엇인가?  

## 2. 배경 및 맥락  
웹 기반 작업을 수행하는 에이전트(agent)는 웹페이지의 문서 객체 모델(DOM)과 화면 스크린샷을 관측한 뒤, 그 상황에서 수행할 행동을 생성한다. WebArena와 같은 환경은 전자상거래, 포럼, 협업 도구 등 실제와 유사한 웹사이트를 제공하여, 에이전트가 고차원 명령(high-level instruction)을 단계별 행동으로 수행하도록 설계되었다【2†L23-L30】. Zhou 등은 WebArena 프로젝트에서 “네 개 도메인의 진짜 웹사이트”를 구성하고, 인간-수준의 장기 계획 능력을 요구하는 작업 셋을 내놓았다【24†L28-L36】【24†L46-L54】. 이들 작업은 실제 사용 사례와 유사한 난이도를 반영하며, GPT-4 기반 에이전트도 약 14.4%의 성공률밖에 달성하지 못하는 반면, 인간은 78.2% 성공률을 보이는 등 여전히 도전적이다【24†L46-L54】. 이처럼 **현실적인 웹 작업**에서는 많은 단계의 결정과 외부 지식 활용이 필수적이며, 이를 학습·평가하려면 풍부한 궤적 데이터가 필요하다. 에이전트 궤적은 일반적으로 시간 순서에 따른 (관측, 행동, 추론) 쌍의 연속으로 정의된다【16†L231-L239】. 각 관측은 DOM 트리와 접근성 트리(accessibility tree), 스크린샷을 포함하며, LLM 에이전트는 이를 입력으로 받아 행동을 추론한다【16†L231-L239】.  

## 3. 문헌 검토  
기존 연구에서는 다양한 웹 에이전트 벤치마크와 데이터셋이 제안되었다. 예를 들어, WebShop(Yao et al. 2022), Mind2Web(Deng et al. 2023) 등 초기 벤치마크는 제한된 도메인 환경에서 룰 기반 평가를 사용했다【1†L160-L169】. 최근 WebArena(Zhou et al. 2024)는 보다 현실적 환경을 제안했고, 이와 연계해 VisualWebArena(Koh et al. 2024)와 VideoWebArena(Jang et al. 2024) 등이 시각·멀티미디어 작업을 확장했다【1†L161-L170】. 또한 AgentRewardBench(Pan et al. 2024)은 LLM 기반 자동 평가를 위해 1300개 이상의 궤적을 수집하여 전문가 레이블을 제공한다【16†L139-L147】. WebGraphEval(Ma et al. 2024)은 여러 에이전트 궤적을 그래프로 통합하여 평가하는 방법을 제안했으며, 궤적을 정규화하고 합치는 **데이터 전처리** 절차를 강조했다【1†L225-L234】【1†L236-L244】. 한편, Xue et al.(2025)은 “Online-Mind2Web” 벤치마크(136개 사이트, 300개 작업)를 도입하며 기존 벤치마크의 한계를 지적했다. 기존 대부분 벤치마크가 **캐시된 정적 페이지**나 제한된 시뮬레이션 환경에서 진행되기 때문에 실제 웹 사용과 차이가 크다고 보고했다【29†L110-L115】. 이들은 실제 사용자 사용 방식과 유사한 환경을 강조하며, 실제 웹사이트에서의 작업을 포함할 것을 제안했다【29†L64-L72】【29†L110-L115】. 이와 같은 문헌들을 종합하면, 웹 에이전트 연구를 위한 궤적 데이터는 *현실성, 다양성, 체계적 전처리*가 중요하다는 점이 밝혀진다.  

## 4. 데이터 출처 및 벤치마크  
현재 웹 에이전트 연구에서 사용되는 데이터 출처는 크게 (1) **사람 혹은 에이전트 궤적 수집**과 (2) **환경-기반 벤치마크**로 나눌 수 있다. 대표적으로 WebArena 팀은 2023년 11월까지 179개 작업의 사람 궤적을 Playwright trace 파일 형태로 공개했다【14†L259-L266】. 각 궤적 파일은 `<task_id>.zip` 형태로 저장되며, 작업 설정(task template)을 통해 추적할 작업이 지정된다. 포괄적인 데이터 접근을 위해 Playwright를 통해 브라우저 상태, 네트워크 이벤트 등 생세한 정보를 기록하며【14†L259-L266】, 이 데이터에서 에이전트의 관측, 행동, 추론을 추출할 수 있다. 예를 들어, WebArena 리소스 설명에 따르면 *“각 파일(render_*.html)은 (1) 접근성 트리 관측, (2) 모델의 원시 예측(raw prediction), (3) 파싱된 액션”*과 스크린샷이 포함되어 있다【14†L298-L305】. 이는 Data 포맷 설계의 좋은 예로, 한 궤적 내에서 필요한 모든 정보가 별도의 HTML 요소로 명시되어 있다(코드 예시 참조【14†L306-L314】).  

또한 WebArena 확장 프로젝트로 WebArena-Infinity 데이터셋이 공개되었는데, 해당 데이터셋은 23,600개 이상의 학습 궤적을 포함하는 대규모 컬렉션이다【9†L66-L70】. 이처럼 규모가 큰 데이터셋은 실험을 위한 충실한 훈련 데이터를 제공한다. Mind2Web나 WebVoyager 같은 연구진도 실제 웹사이트를 활용한 궤적 데이터를 구축했으며【29†L64-L72】, OpenAI Operator나 Claude Computer Use 등의 상업적 에이전트 역시 내부적으로 방대한 웹 상호작용 데이터를 수집하는 것으로 알려져 있다. 요약하자면, 웹 에이전트 데이터 출처는 (가) WebArena 등 합성 환경과 (나) 실제 웹 태스크 환경 모두에서 궤적을 수집하는 형태로 구성된다. 이들 출처의 데이터를 종합하면, 다양한 도메인(쇼핑, 포럼, 지도 등)을 아우르고 에이전트의 행동-추론이 모두 포함된 풍부한 데이터를 확보할 수 있다.  

## 5. 궤적 데이터 구조 (관측-행동-추론)  
【23†embed_image】WebArena와 유사한 환경에서는 각 단계마다 브라우저의 **DOM 트리**와 **스크린샷** 같은 관측 정보, 에이전트의 원시 추론 텍스트, 그리고 실행된 행동(action) 정보를 한 쌍으로 저장한다. 위 그림은 WebArena 실행 중 Playwright가 기록한 궤적 뷰어 화면 예시이다. 각 시점의 화면 상태와 함께 좌측 패널에는 이벤트 타임라인이 표시되며, 관측 예로는 `<div class="state_obv">` 요소에 접근성 트리가, 행동 예로는 `<div class="action_object">`에 클릭/타이핑 명령이 텍스트 형태로 기록된다【14†L298-L305】【14†L306-L314】. 이처럼 **에이전트 궤적 데이터는 (관측 – 추론 – 행동) 삼중 구조**를 따라야 한다. 대표적인 형식 정의로, AgentRewardBench에서는 궤적을 관측(observation), 행동(action), 추론(reasoning)의 순차열로 정의했고, 각 관측에 스크린샷·DOM·접근성 트리를 포함시켰다【16†L231-L239】. LLM 에이전트가 관측을 입력받아 추론을 생성하고(action 전에), 이를 파싱해 행동을 얻는 과정이 명확히 기록되어야 한다. 실제 WebArena 데이터에서도 각각의 `render_*.html` 파일에 (1) 관측 텍스트, (2) 모델의 출력 문장, (3) 파싱된 action 객체가 포함되어 있어, 추후 스크레이핑으로 쉽게 정보 추출이 가능하다【14†L298-L305】【14†L306-L314】. 

## 6. 데이터 수집 방법  
웹 에이전트 궤적을 수집하기 위해서는 실험 환경 설정과 데이터 로깅이 핵심이다. WebArena의 경우, 로컬에 호스트된 웹사이트(쇼핑, 포럼 등)에서 에이전트를 실행하며, Playwright를 사용해 사용자의 브라우저 조작과 페이지 렌더링을 기록한다【14†L259-L266】. 연구자는 사람이 직접 또는 LLM 에이전트를 운영해 목표 작업을 수행하도록 한 뒤, 각 행동 단계의 DOM과 스크린샷을 저장한다. 이 과정에서 **메타데이터**도 함께 수집할 수 있는데, 예를 들어 작업 ID, 응답 시간, 성공/실패 여부 등이 있다. AgentRewardBench처럼 복수 에이전트로부터 궤적을 얻을 때는 공통 작업 분류(baselined tasks)를 사용하여 서로 다른 모델들의 궤적을 비교 저장한다【16†L139-L147】. 데이터 규모는 수백~수천 궤적으로 구성되며, AgentRewardBench의 경우 LLM 4종으로부터 1300여 개 궤적을 수집하여 레이블링했다【16†L139-L147】. 뿐만 아니라, Mind2Web(오하이오주립대) 등은 실제 웹사이트를 대상으로 작업 목록을 정의해 온라인으로 에이전트들을 평가하며, 실제 사용자 사용 방식과 유사한 데이터셋을 구축한다【29†L64-L72】【29†L110-L115】. 이처럼 데이터 수집 단계에서는 (1) 풍부하고 다양한 작업 설정, (2) 에이전트 행동 기록, (3) 메타정보 기록의 세 가지 요소를 모두 체계적으로 설계해야 한다.  

## 7. 데이터 품질 이슈  
수집된 웹 에이전트 데이터는 여러 품질 이슈에 주의해야 한다. 첫째, **다양성 부족** 우려가 있다. 제한된 도메인, 유사한 작업 템플릿만 사용하면 데이터가 편향될 수 있다. Zhou et al.도 WebArena 작업이 실제 생활 작업과 유사하도록 고안했으나, 여전히 실세계 웹의 복잡성을 완벽히 반영하지는 못한다【24†L28-L36】. Xue 등은 많은 벤치마크가 정적 HTML 스냅샷이나 제한적 시뮬레이션 사이트에 의존한다고 지적했는데【29†L110-L115】, 이는 실제 웹브라우징의 다양성을 저해한다. 따라서 가능한 **실제 웹페이지**(가능하면 다양한 인기 사이트)로 작업 범위를 확장하거나, WebArena와 같은 동적 도구들을 적극 활용해야 한다.  

둘째, **노이즈와 오류** 문제다. 자동 에이전트는 때때로 원치 않는 행동이나 예측을 할 수 있으며, 기록 과정에서도 로그 누락이나 잘못된 캡처가 일어날 수 있다. 예를 들어, 네트워크 오류로 인한 페이지 로딩 실패가 있을 수 있다. 이러한 노이즈를 걸러내지 않으면 학습 시 왜곡이 생긴다. 따라서 데이터 수집 후 검증 단계에서 불완전 궤적을 제거하거나 보완해야 한다.  

셋째, **라벨링 일관성** 이슈가 있다. 성공/실패 여부나 사이드 이펙트 등의 레이블은 사람에 따라 달라질 수 있다. AgentRewardBench는 전문가가 각 궤적을 리뷰하여 성공 여부, 부수 효과 여부, 반복 사이클 포함 여부 등을 라벨링했는데【16†L139-L147】, 이들은 LLM 평가보다 인간 기준을 강조했다. 단순 룰(rule-based)만 사용할 경우 실제 성공 케이스를 놓칠 위험이 있다【16†L143-L147】. 즉, 높은 품질의 라벨을 확보하려면 사람이 검증하거나 LLM judge를 활용하여 인간 기준과 최대한 일치시키는 방법을 써야 한다【16†L143-L147】【1†L236-L240】.  

마지막으로, **개인정보 및 안전** 측면도 고려해야 한다. 실제 웹 데이터를 사용할 때 개인정보 노출이나 안전하지 않은 내용이 포함될 수 있다. 특히 프라이빗 사이트나 악성 컨텐츠가 궤적에 들어갈 경우, 데이터 처리 과정에서 이를 제거하는 필터링이 필요하다. 

## 8. 전처리 설계 (Controlled Experiments 준비)  
제어된 실험을 위해 데이터 전처리는 매우 중요하다. 먼저 **데이터 정규화** 단계에서 행동(action)들을 표준 형식으로 변환해야 한다. WebGraphEval 예시처럼, 서로 다른 자연어 표현으로 같은 동작을 나타내는 경우 LLM을 사용해 이를 일관된 액션 단위로 매핑한다(예: “Submit 버튼 클릭”과 “Press Submit”을 `click(text='Submit')`로 통일)【1†L227-L234】. 정규화하지 않으면 동일 행위가 서로 다른 노드로 취급되어 분석이 어려워진다.  

다음으로 **성공/실패 라벨링**을 자동화하고 검증하는 것이 필요하다. LLM-as-Judge 프로토콜을 사용하여 각 궤적의 최종 상태가 작업 목표를 만족하는지 평가할 수 있다【1†L236-L240】. 이때 단순한 규칙(rule-based)만 쓰면 특히 성공률이 과소평가되는 경향이 있으므로【16†L143-L147】, LLM 판단을 보조적으로 활용하거나 인간 리뷰를 병행하는 것이 권장된다.  

또한 **불필요하거나 중복된 행동 제거**도 중요하다. WebGraphEval 파이프라인에서는 각 액션의 필요 여부(necessity)를 LLM으로 판단해, 같은 목적을 위해 반복된 동작을 식별하고 제거한다【1†L242-L244】. 이는 궤적을 간소화하고 모델 학습 시 잡음을 줄인다.  

데이터 전처리 과정은 아래 흐름도로 요약할 수 있다:  
```mermaid
flowchart LR
    A[환경(WebArena 등)] --> B[궤적 수집 및 로깅]
    B --> C[원시 데이터 파싱 (DOM, 스크린샷, 모델 출력 추출)]
    C --> D[행동 정규화 (LLM 기반 일관된 명세 부여)]
    D --> E[성공/실패 라벨링 (LLM·규칙·인간 평가)]
    E --> F[데이터 분할 (훈련/검증/테스트)]
    F --> G[학습 및 평가 준비 완료]
```  
위 흐름도에서 알 수 있듯, 궤적 데이터는 수집→파싱→정규화→라벨링→분할 과정을 거쳐야 한다. 특히 분할 단계에서는 동일한 작업이 훈련과 평가 세트에 중복되지 않도록 해야 한다. AgentRewardBench는 1302개 궤적을 196개(dev)와 1106개(test)로 나누었는데, 이런 작업 기반 분할을 통해 과적합을 줄이고 일반화 성능을 확보했다【30†L7-L10】.  

## 9. 통제된 실험 설계  
통제된 실험을 위해 데이터셋을 설계할 때는 변수조절과 반복실험이 가능하도록 신경써야 한다. 예를 들어, 원하지 않는 변수를 통제하기 위해 (1) 작업 난이도나 종류를 고정하고 다른 부분만 변경하거나, (2) 동일 작업에 학습용/평가용 에이전트를 다르게 적용하는 방법을 사용할 수 있다. Prompt injection 실험의 경우라면, (3) 일부 궤적에만 의도된 악성 페이로드를 삽입하는 등 *대조군 vs 실험군*을 명확히 구분해야 한다. 이때 전처리 단계에서 삽입된 콘텐츠를 정확히 추적·표시하여 평가에 반영해야 한다.  

또한 데이터 준비 시 랜덤성 통제에도 주의한다. 충분한 셔플과 교차 검증을 통해 훈련/검증 구성이 편향되지 않도록 한다. 일반적으로 *에포크 간 훈련 데이터 순서를 무작위화*하여 특정 경로에 의존하는 모델 학습을 방지하고, *k-겹 교차 검증* 등을 통해 결과의 안정성을 점검할 수 있다.  

마지막으로 데이터 증강(Data Augmentation) 기법도 고려할 수 있다. 예를 들어, 작업 목표 문장만 일부 바꾸거나, 에이전트 대화(log)를 다양한 방언/문체로 변환하여 모델의 일반화 능력을 평가할 수 있다. 이는 데이터 부족을 보완하고, 모델이 특정 표현에 과도하게 민감해지지 않도록 도와준다.  

## 10. 일반적 실수 및 주의사항  
웹 에이전트 데이터 준비에서 자주 발생하는 실수를 경고하자면 다음과 같다. 

- **데이터 누수(Leakage)**: 훈련/평가 데이터를 무작위로 분할할 경우 동일 작업이나 페이지가 양쪽 세트에 섞여 들어가면, 평가 결과가 과장된다. 이론적으로 작업 또는 웹사이트 단위로 분리해야 한다. AgentRewardBench처럼 작업 단위로 분할하면【30†L7-L10】 보다 엄격한 평가가 가능하다.  
- **불일치한 전처리**: 훈련과 평가 데이터에 서로 다른 전처리 규칙을 적용하면 비교가 불가능해진다. 예를 들어, 훈련 시에는 HTML 주석을 제거했지만 평가 시에는 하지 않는다면, 모델의 입력이 달라져 예측에 오류가 생길 수 있다. 따라서 모든 데이터에 동일한 전처리 파이프라인을 적용해야 한다.  
- **부적절한 라벨링**: 평가 기준을 명확히 하지 않으면 라벨 편향이 생긴다. 예를 들어 성공 여부를 단순히 목표 텍스트 포함 여부로만 판단하면, 비표준 성공 사례가 누락될 수 있다【16†L143-L147】. 반면 LLM을 통한 자동 평가만 쓰면 중요한 오류를 놓칠 수 있으므로【16†L143-L147】【1†L236-L240】, 가능한 한 사람이 검토하거나 복수의 평가자 간 합의를 고려해야 한다.  
- **데이터 오염**: 웹 크롤링 시 개인정보나 외부 링크가 무분별하게 포함될 수 있다. 특히 에이전트가 로그인 정보나 사용자 데이터에 접근하는 작업에서는 주의가 필요하다. 필터링 단계에서 개인정보(예: 이메일, 주소)나 보안 위험이 있는 액션(예: 관리자 페이지 접근)을 제거해야 한다.  
- **과도한 복잡화**: 전처리에서 너무 많은 기법을 한꺼번에 쓰면 실험 재현성이 떨어질 수 있다. 예를 들어, 모든 궤적을 과도하게 압축하거나 불필요한 정보를 삭제하면, 나중에 결과를 재구성하기 어려워진다. 실험 기록을 남기고, 단계별로 데이터를 버전관리하여 변경 내역을 추적해야 한다.  

## 11. 연구 동향과 통합 결과  
- **합의된 관점**: 주요 연구들은 웹 에이전트의 궤적 데이터를 체계적으로 기록해야 한다는 점에 동의한다. 특히 WebArena와 AgentRewardBench 모두 (*관측, 행동, 추론*)의 삼중 구조를 강조했다【14†L298-L305】【16†L231-L239】. 또한 대규모 데이터셋 구축과 라벨 품질 확보가 필수적임이 입증되었다【16†L139-L147】【9†L66-L70】. 최근 트렌드는 *LLM judge*를 활용한 자동 라벨링과 그래프 기반 분석으로, 이는 과거 룰 기반 평가의 한계를 보완하는 방향으로 발전 중이다【16†L143-L147】【1†L227-L234】.  
- **이견 및 한계**: 다만 데이터의 현실성 측면에서는 연구자들 사이에 차이가 있다. Xue 등은 실시간 웹 평가를 강조하며 실세계 웹사이트를 활용해야 한다고 주장한 반면【29†L110-L115】, 일부 벤치마크는 여전히 에뮬레이터 기반 작업이나 통제된 환경을 사용한다. 데이터 품질이나 다양성 측면에서, 현재 공개된 벤치마크들은 실제 웹의 변동성을 완벽히 반영하지 못할 수 있다는 지적이 있다.  
- **미충족 영역**: 궤적 데이터의 프라이버시 보호, 대규모 온라인 실험 데이터, 다국어·비기능성 웹사이트 데이터 등은 아직 충분히 다뤄지지 않았다. 또한 궤적 생성 모델을 활용한 합성 데이터(데이터 합성) 기법은 연구 초기 단계이며, 그 품질 검증이 필요하다.  

## 12. 시사점 및 제언  
- **데이터 설계 시 현실성 강조**: 연구 실험을 실제 웹 사용 환경과 가깝게 구성해야 한다. WebArena처럼 풍부한 도메인과 실제 데이터 구조를 모방하거나, Mind2Web 처럼 실제 사이트 기반 작업을 사용해 데이터 현실감을 높일 것을 권장한다【24†L28-L36】【29†L64-L72】.  
- **전처리 자동화·표준화**: 행동 정규화, LLM 라벨링, 중복 제거 등의 전처리 파이프라인을 미리 구축하고 자동화해야 한다. WebGraphEval에서는 이러한 과정을 LLM 기반으로 자동화하여 분석 용이성을 높였다【1†L227-L234】【1†L236-L240】. 이와 같이 신뢰할 수 있는 자동화 도구를 활용하면 대규모 데이터 처리에 소요되는 시간을 줄일 수 있다.  
- **교차 검증 강화**: 모델 학습 시 데이터 분할과 검증을 엄격히 하여 일반화 능력을 확인해야 한다. 예를 들어 동일한 작업에서 훈련/평가 데이터가 중복되지 않도록 작업 단위로 분할하고, 필요시 k-겹 교차 검증을 수행한다. 이는 데이터 누수로 인한 성능 왜곡을 방지한다【30†L7-L10】.  
- **데이터 공유 및 재현성**: 연구 커뮤니티 전반에서 데이터셋과 코드, 처리 파이프라인을 공개·표준화하면 장기적으로 연구 품질이 향상된다. WebArena 사례처럼 데이터 형식을 공개하고 사용법을 안내【14†L259-L266】【14†L298-L305】하는 것이 좋은 모범이다.  

## 13. 한계점 및 미래 연구  
본 분석은 웹 에이전트 데이터 설계 관점에 집중했으나, 실제 적용 환경에서는 여전히 미해결 문제가 많다. 예를 들어, **유저 간 변동성**(개개인마다 웹 탐색 패턴이 다름)이나 **온라인 업데이트**(웹페이지의 지속적 변화)에 대한 대응이 어렵다. 또한 자원 제한(계산·저장) 하에서 대규모 궤적을 효과적으로 관리하는 방법은 추가 연구가 필요하다. 메모리 소모가 큰 스크린샷 대신 지표(feature)로 대체하는 등 경량화 방안도 고민해야 한다.  

게다가, 본 보고서는 공개 자료와 문헌에 기반해 일반적인 권장사항을 제시했으므로, 특정 시나리오(예: 특정 웹사이트군, 특정 LLM 모델 종류)에서 최적화된 솔루션이 아닐 수 있다. 각 실험에서는 본 분석을 기반으로 자신의 데이터 특성을 면밀히 검토하여 맞춤형 설계를 해야 할 것이다.  

## 14. 향후 연구 방향 및 참고 자료  
향후 연구로는 (가) 더욱 다양한 실제 웹사이트와 사용자 행동을 반영한 대규모 궤적 데이터 구축, (나) LLM judge 정확도 향상을 위한 멀티모달 데이터 활용, (다) 악의적 삽입 공격 데이터셋 구축 등이 유망하다. 또한 웹 에이전트 데이터와 관련된 인터페이스(RL 설정, RAG 시스템 등) 연구와 연계하여, 메모리 저장·요약·검색 등 전반적 데이터 파이프라인을 통합적으로 고도화하는 작업도 필요하다.

## 주요 시사점  
- **궤적 구조:** 웹 에이전트 궤적은 관측(observation), 행동(action), 추론(reasoning)의 연속으로 구성되며, 각 관측에 스크린샷과 DOM/접근성 트리 정보를 포함해야 한다【14†L298-L305】【16†L231-L239】.  
- **데이터 출처:** WebArena 등의 합성 환경에서 수집된 수백~수천 개의 궤적 데이터(예: 179개 작업×여러 실험자)와, Live 웹 기반 벤치마크(예: Online-Mind2Web)의 데이터가 활용된다【14†L259-L266】【29†L64-L72】.  
- **전처리:** 액션 정규화 및 성공/실패 라벨링이 필수적이다. 자연어 행동 표현을 LLM으로 표준화하고【1†L227-L234】, 궤적마다 LLM judge를 통해 성공 여부를 평가해야 한다【1†L236-L240】.  
- **실험 설계:** 훈련/평가 데이터 분할은 작업 단위로 수행하여 과적합을 방지해야 하며【30†L7-L10】, 전처리 과정에서 데이터 누수나 편향이 없도록 주의해야 한다.  
- **제언:** 가능하면 데이터와 파이프라인을 공개하고, 다양한 출처의 데이터를 수집하여 신뢰성과 일반화를 확보해야 한다.
